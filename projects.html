TTS 프로젝트

Period: 2025.09 ~ Present

Role: PM 및 풀스택 개발

배경

고객 상담 센터에서 실시간 음성 합성이 필요했고, 기존 시스템의 응답 시간(2.3초) 및 동시 처리 채널(12개) 제약 극복 필요. 음성 품질도 중요.

구현

시스템 아키텍처: FastAPI WebSocket/REST 게이트웨이 + vLLM-Omni(Qwen3-TTS) 추론 엔진 + Redis 분산락 & 캐싱 + React 대시보드. 문장 단위 논리 청킹 및 WebSocket 스트리밍으로 TTFB 단축.

성능 최적화: H100/L40S GPU 추론 환경 구성, FP8 양자화, executor backend 최적화 및 concurrency tuning으로 추론 처리량 개선. Pre-Normalization, CUDA Graph Warmup, 세마포어 동시성 제어로 GPU 메모리 활용 개선. Redis 캐싱으로 동일 문장 0ms 응답 구현. 5가지 엔지니어링 문제 해결 (팝 노이즈, GC 누수, 세션 탈취, ITN 토크나이저, CPU 병목).

기능 확장: 참조 음성 5초만으로 화자 목소리를 복제하는 ICL(In-Context Learning) 기반 음성복제 기능 구현. OpenAI TTS API 호환 인터페이스와 기존 WebSocket 프로토콜 호환 레이어를 만들어 외부 서비스가 코드 수정 없이 연동되도록 지원. Redis 기반 Atomic quota, 분산락, Circuit Breaker 등 장애 대응 로직 개발.

모델 및 품질: TTS 학습 데이터 정제 및 ZipVoice 모델 학습. Whisper 기반 CER/WER 자동 품질 측정 파이프라인 구축, 사내 운영 테스트 및 QA 수행.

성과 (KPI)

  • TTFB: 2292ms → 334ms (85% 단축, 6.85배 개선)
  • Throughput: 0.68 → 1.41 rps (2배)
  • 최적화 후: ~1.7 rps (2.5배 향상)
  • 동시 처리 채널: 12 → 24 (2배)
  • 단건 지연시간 15% 감소
  • RTF (Real-Time Factor): 0.1 이하
  • CER(문자 오류율): 평균 3~5% (Whisper-large-v3 STT 기반 자동 품질 측정)
  • 32개 언어 지원 단일 플랫폼
  • ITN 정규화 정확도 50.1% → 70.9%

핵심 엔지니어링 (6가지 문제 해결)

  1. 스트리밍 팝 노이즈 — 스트리밍 중 간헐적으로 '틱' 잡음 발생. PCM 청크가 홀수 바이트로 끊겨 와 16bit 샘플 정렬이 깨지는 것이 원인. 잔여 바이트를 다음 청크에 이어 붙이는 Residual Buffer로 정렬을 보장하고, 청크 경계에 Micro-fade를 적용해 해결
  2. 스토리지 누수 — 보이스를 삭제해도 디스크 사용량이 계속 증가. 원본을 지워도 파생된 MD5 해시 캐시 파일이 남는 구조여서, 파생 파일까지 추적해 함께 지우는 Deep GC를 구현해 차단
  3. 웹소켓 세션 탈취 방어 — 비밀번호를 바꿔도 이미 발급된 토큰으로 웹소켓 연결이 열리는 허점 발견. 토큰 발급 시점과 password_changed_at을 대조해 변경 이전 토큰을 전부 무효화
  4. ITN 정규화 — URL·이메일을 한국어로 부자연스럽게 읽는 문제. 자연스러운 발음으로 변환하는 스마트 ITN 엔진을 직접 개발해 정규화 정확도 50.1% → 70.9%
  5. CPU 병목 — 요청마다 ThreadPoolExecutor가 새로 생성되는 오버헤드를 프로파일링으로 발견. 이를 걷어내고 텍스트 처리 로직을 O(N²) → O(1)로 개선
  6. SSE 스트리밍 호환 — ZipVoice 엔진이 첫 청크에만 WAV 헤더를 포함해 브라우저 decodeAudioData()가 두 번째 청크부터 실패. 원인을 분석해 청크별 헤더를 수정하고 실시간 SSE 엔드포인트를 신설, 데모 페이지와 API 문서까지 제공해 해결

성과

  • 회사 최초 Kubernetes 기반 AI 플랫폼 설계 및 구축
  • MetalLB 로드 밸런싱, Ingress-nginx 라우팅, PersistentVolume 스토리지 통합
  • GitLab → Jenkins → ArgoCD 완전 자동화 CI/CD 파이프라인 (GitOps 기반)
  • Harbor 중앙 레지스트리로 Docker 이미지 관리
  • Prometheus/Grafana/Loki/Kibana 기반 모니터링 및 로그 자동화

배경

LLM Agent 시스템(RAG, tool orchestration, state 관리)의 E2E 구축 역량 확보 목적. 포트폴리오 기술문서를 지식 베이스로 하는 Q&A Agent를 전부 로컬 환경(Ollama)에서 구현.

구현

Agent 실행 흐름: LangGraph StateGraph로 retrieve → grade(검색 품질 자가 평가) → rewrite(질문 재작성) → generate 흐름 설계. 검색 결과가 부족하면 질문을 재작성해 재검색하는 self-corrective 루프 구현.

RAG 파이프라인: 문서 청킹 → BGE-M3 임베딩 → FAISS + BM25 하이브리드 검색 (RRF 직접 구현). 문서 근거 기반 답변 + 출처 인용, 근거 없으면 답변 거부(hallucination 억제). FastAPI 서빙 레이어 포함.

Tool Orchestration: LLM이 도구(검색/계산/날짜)를 선택 호출하는 multi-step Agent 레이어. 소형 모델의 tool calling 무응답을 감지해 검색을 강제 호출하는 결정적 폴백 설계.

Multi-Agent: 멀티홉 질문을 Planner가 사실 단위 sub-질문으로 분해 → RAG Worker 병렬 처리 → Synthesizer 종합하는 계층 구조. Planner가 이미 검색용으로 정제한 질문이므로 Worker의 자체 재작성 루프는 비활성화해 검색 교란을 방지.

소형 모델 디버깅 (통제 실험): temperature를 0으로 고정해 실패를 재현 가능하게 만든 뒤 조건을 하나씩 바꿔가며 원인을 격리. 청크 순서 실험으로 3B 모델의 lost-in-the-middle 특성을 확인해 최상위 청크를 질문 바로 앞에 배치했고, "(2025.09~)" 같은 압축 날짜 표기를 읽지 못하는 문제는 표기 규칙 힌트로, "~의 시작 기간" 같은 명사형 질문에서만 답변을 거부하는 조건은 프롬프트 격리 실험으로 찾아 Planner가 의문문을 생성하도록 수정.

평가 루프 및 개선 (실측)

  • 10문항 평가셋으로 정답률/재작성률/지연시간 측정 파이프라인 구축

  • 정답률 50% → 60%, 재작성률 100% → 30% (개선 전 과정 git 히스토리에 기록)
